我的第一篇正式博客：从零搭建 Hexo 并部署到 GitHub Pages

这是我的第一篇正式博客。我会记录使用 Hexo 搭建个人博客、部署到 GitHub Pages 的完整过程，也把后续写文章时常用的 Markdown 排版、图片、代码和流程图整理到一起。

内容包含几个很容易踩到的坑：账号名和显示名的区别、网络连接、推送失败，以及如何免费更换主题。

一、准备环境

开始前需要准备以下内容：

  • 一个 GitHub 账号，用于存放博客和发布网站。
  • Node.js LTS 版本。安装后在终端执行 node -v、npm -v，能显示版本号就说明安装成功。
  • Git。执行 git --version 检查是否可用。
  • 一个稳定的网络环境，能够正常访问 GitHub 和 npm。

Windows 用户建议使用 PowerShell、CMD 或 Git Bash。若 PowerShell 执行 npm 时出现“禁止运行脚本”之类的提示，可以直接使用 npm.cmd，例如：

三、创建 GitHub 仓库和 Pages

GitHub Pages 有两种常见地址形式，仓库名称要和使用方式匹配：

类型 仓库名称 访问地址
个人主页 你的用户名.github.io https://你的用户名.github.io/
项目主页 任意仓库名，例如 blog https://你的用户名.github.io/blog/

如果要使用最简洁的个人主页地址，仓库必须叫作 你的用户名.github.io。例如用户名是 afterTvT，仓库就应是 afterTvT.github.io。

创建仓库时建议选择公开仓库。创建完成后，在仓库的 Settings -> Pages 中选择：

  • Source：Deploy from a branch
  • Branch：main
  • Folder：/(root)

保存后，GitHub 会开始发布网站。首次发布通常需要等待几分钟。

GitHub 已不支持用账号密码进行 HTTPS 推送。如果弹出登录窗口，按提示通过浏览器完成 GitHub 授权即可。也可以改用 SSH 地址，但需要先把本机 SSH 公钥添加到 GitHub 的 Settings -> SSH and GPG keys。

七、最容易出错的地方

1. 把显示名当成用户名

GitHub 个人资料里的 Name 只是显示名称，不是登录用户名。仓库地址左上角显示的账号名才是真正的用户名。例如显示名可以叫 after，真实用户名仍可能是 afterTvT。部署地址必须使用真实用户名。

2. 出现 Repository not found

常见原因有三个：仓库 URL 拼错、使用了显示名而非用户名、当前 Git 没有该私有仓库的权限。优先检查 _config.yml 中的 repo，然后在浏览器中打开同一个仓库地址确认它存在。

3. 出现 rejected (fetch first)

这说明远程仓库已有比本地更新的提交，例如创建仓库时自动生成的 README。不要强制覆盖不确定的远程内容。先拉取并查看差异，再将本地部署记录合并或变基后推送。

4. 页面打开后样式丢失或图片 404

多数是 url 和 root 写错。个人主页使用 root: /；项目主页需要写成 root: /仓库名/。修改后重新执行构建和部署。

5. GitHub 有提交，网站却没有更新

检查仓库 Settings -> Pages 是否选中了 main 分支和 /(root)。GitHub Pages 有缓存和构建时间，通常等待几分钟后再刷新页面即可。

九、日常使用清单

日常写博客其实只需要记住四件事：

  1. hexo new "文章标题" 创建文章。
  2. 在 source/_posts 中用 Markdown 写内容。
  3. hexo server 本地检查效果。
  4. npm.cmd run build 和 npm.cmd run deploy 发布到 GitHub。

保持文章源文件、主题配置和 GitHub 仓库地址的备份，博客就可以长期稳定地维护下去。

十三、发布前检查

每次发布前，可以按下面的顺序快速检查：

  1. 检查文章标题、日期、分类和标签。
  2. 检查图片路径，确保本地预览时图片能显示。
  3. 执行 hexo server，在浏览器检查排版、链接和代码块。
  4. 执行 hexo clean、hexo g、hexo d 发布。
  5. 等待 GitHub Pages 构建完成，再打开线上页面确认。

写博客不必一开始就追求非常复杂的排版。先把问题、过程、结果和踩坑经验说清楚，持续积累几篇后，博客自然会慢慢形成自己的风格。
